HELP

Overview

Parsival ingests messages from Outlook, Slack, GitHub, Jira, and Microsoft Teams, runs each one through a local LLM, and surfaces what needs your attention. Every item gets a priority, category, action items, and a project tag.

All analysis runs locally via Ollama — no data leaves your infrastructure.

Topbar buttons

ButtonWhat it does
▶ ScanFetch new items from all enabled sources (Outlook, Slack, GitHub, Jira, Teams) and run LLM analysis on anything new. Turns into ■ Stop while running.
↺ Re-analyzeRe-run LLM analysis on every stored item using your current project settings. Use this after adding projects or changing keywords. Turns into ■ Stop while running.
◈ SeedBootstrap project intelligence from your existing data — the LLM reads your stored items and proposes project names, keywords, and watch topics you can review and apply before running a full analysis.

⚙Settings — configure your identity, Ollama endpoint, projects, connectors, and credentials.

Source toggles

The pill buttons (Outlook, Slack, GitHub, Jira, Teams) in the topbar filter which sources are shown in the item list. They also control which sources are included in the next scan — only active (lit) sources are scanned.

Left sidebar — filters

FilterMeaning
All / User / Project / Topic / GeneralHierarchy tier. User = directly addressed to you. Project = related to a configured project. Topic = matches a watch topic. General = everything else.
PassdownsShift handoff notes detected by subject or body pattern.
Projects listClick a project name to filter to its items. Click again to clear. The count next to each name reflects the current filter.

Item list

Items are sorted by priority (high first), then by timestamp. Click any row to open the detail panel. Each row shows priority, category, title with project badge, source icon, and timestamp.

◈
Tag the item to a project. Opens a picker with all configured projects. Select a project to tag; the system learns keywords and senders from the item in the background.
✕ noise
Mark the item as noise. Sets category to noise/low and teaches the noise filter its keywords so similar items are deprioritised in future scans.

Contacts

The Contacts tab is a long-lived directory of every person Squire has seen across email headers — vendors, clients, and colleagues. Identity is a stable internal ID, not an email address, so the record outlives any single field when someone changes employer or address. Multiple emails can be attached to one contact.

Auto-population
Every analyzed item's From, To, and CC fields are scraped automatically. Re-running scans does not duplicate rows — emails are unique across the table and seeing one again just bumps its source count.
Rebuild from items
Rescan every existing item's headers and refresh the contacts table. Idempotent, so safe to run after a schema change or a bulk import.
Re-parse signatures
Walks every email body and tries to extract phone, title, employer, and address from the sender's signature block. Each contact field is tagged with its provenance — header (auto from To/CC), signature (parsed from the email body, with a confidence %), or manual (you edited it). Anything you've edited by hand is locked from automatic re-parses.
Manual editing
Every field is editable: name, phone, employer, title, employer address, notes, and email list. Click any row to open the editor; + New contact creates one from scratch. Editing a field flips its tag to manual, which stops the signature parser from ever overwriting it.
Owner resolution
When the LLM extracts a delegated action item like "Mike, pull the drawings" and Mike is not one of the email's recipients, the action item's assignee is filled in from the contacts table.
Email merging
Adding an email already attached to a different contact returns an error — consolidate manually by editing both rows. The primary email is the first one in the list.
Signature parsing (extracting phone numbers and titles from email body footers) is tracked separately and not part of this feature.

Situations

Situations are cross-source groupings of related items that the system identifies as part of the same operational thread — for example, an email thread, a Slack conversation, and a Jira ticket all about the same issue. Each situation has a lifecycle status, a composite urgency score, optional follow-up date, and a deduplicated action list.

Status workflow
new → investigating → waiting (with optional follow-up date) → resolved or dismissed. Use the status dropdown on each situation card.
Follow-up date
Set a date when the situation needs attention next. Situations in "waiting" status past their follow-up date surface at the top of the list with a visual indicator.
Score
Composite urgency — higher means more attention needed now.
Action items
Each situation card lists your action items inline. Completed items appear with a strikethrough and checkmark so you can see progress at a glance.
Default filter
The list shows new, investigating, and waiting situations by default. Toggle to see resolved/dismissed.
Transition log
Every status change is logged with timestamp and optional note.

Projects

Projects are named workstreams the LLM uses to tag and group items. Configure them in Settings.

Name
Exact name used for tagging. The LLM will only assign this name — it will not invent variations.
Parent
Optional parent project. When set, the LLM prefers this sub-project over its parent when both could match.
Description
Free-text scope, location, or purpose. Shown to the LLM to help it classify ambiguous items.
Keywords
Comma-separated terms. Any item whose title or body contains these terms is a candidate for this project.
Senders
Comma-separated email addresses or display names. Items from these senders are tagged to this project deterministically before the LLM runs.
After changing project settings, run ↺ Re-analyze to re-tag all stored items with the updated config.

Project learning

When you manually tag an item to a project (via ◈), Parsival extracts keywords from the item in the background and adds them to the project's learned keyword list. It also records the sender's address in the project's learned senders list. Both feed into future scans automatically.

You can inspect and remove learned keywords and senders from the project row in Settings → Projects (click inspect).

Passdown generator

Click 📋 Passdown above the action list to open the passdown modal. Parsival assembles a shift-handoff email from recent activity — open action items, active situations, upcoming deadlines, recent high-priority items, and items you've replied to.

The left pane is a live preview; the right pane is editable HTML. Tune the look-back window (6 h – 7 days), edit freely, then use Copy HTML to paste into Outlook or Copy text for plain-text channels. Nothing is sent automatically — you own the final message.

Priority override feedback

When you change an item's priority in the detail panel, Parsival prompts for a one-click reason: this person matters, this topic is hot right now, deadline is real, or other. The override is stored and injected into future analysis prompts as a learned rule, so the LLM can weight similar items correctly next time. Pick Skip to change the priority without recording a reason.

Split / merge situations

The correlator occasionally lumps together items that shouldn't be one situation, or splits threads that belong together. Two buttons in each situation card let you fix it by hand:

⎇ Split
Open a checklist of the current items, tick the ones that belong in a brand-new situation, and optionally give it a title. The source keeps the rest; both are lightweight-rescored without re-running the LLM.
⛒ Merge
Pick another active situation; its items are absorbed into this one and the source is dismissed with reason merged_into for auditability.

Use ↺ Resynth afterwards if you want the LLM to regenerate titles and summaries.

Stale situation flag

Situations in waiting with no activity for 7 days, or in investigating for 14 days, get an amber stale badge. This is a nudge to either transition the lifecycle (to resolved/dismissed) or take action — Parsival won't auto-close anything.

Scheduled auto-scans

Configure automatic scanning per connector in Settings → Schedule. Each source can be set to scan at a fixed interval (off / 15m / 30m / 1h / 2h / 4h). Auto-scans run the same analysis pipeline as manual scans. If a scan is already running, the scheduled one is skipped.

Check schedule status in the scan status area — it shows the next run time and last run time for each scheduled source.

Pre-scan noise filters

Noise filters skip known-irrelevant items before the LLM runs, saving inference time. Filtered items are stored as "filtered" (auditable) but skip the analysis pipeline.

Rule types
sender_contains (e.g. "noreply@"), subject_contains (e.g. "Out of Office"), source_repo (GitHub repo name), distribution_list (email alias).
Create from noise
When marking an item as noise, you can choose "Create filter from this?" to auto-fill a rule from the item's properties.
Manage
Settings → Filters shows all rules with the count of items filtered in the last scan.

Adaptive attention model

Parsival learns from your behavior to surface items you're most likely to care about. It tracks actions like opening items, tagging projects, marking noise, and changing situation status — then uses embedding similarity to predict attention for new items.

Sorting
Items are sorted by attention score by default. Switch to chronological or LLM-priority via the sort control.
Visual weight
High-attention items have a bolder border — no numbers or badges to add noise.
Cold start
Until 50 actions are recorded, the system uses the LLM's priority tier and shows "Learning your attention patterns — prioritization will improve as you use the tool."
Adaptation
The model adapts as your focus shifts — actions older than 30 days contribute at 50% weight, older than 60 days at 25%. No configuration needed.

Analysis provider

Choose which LLM runs Parsival's analysis pipeline. By default, prompts go to the local Ollama instance via merLLM. You can switch to a paid Ollama cloud account or Anthropic's Claude API for higher-quality analysis at the cost of an API key.

Ollama (local)
Default. Uses the Ollama Model from the settings section above, routed through the merLLM proxy. Free, runs on your GPUs.
Ollama Cloud
Paid Ollama API. Set the API URL and key. Uses the same prompt format as local Ollama.
Claude
Anthropic Claude API. Set the API key and optionally override the model (default: claude-sonnet-4-20250514). Higher quality analysis.
Model Override
If blank, the Ollama Model field is used. Set this to use a different model for analysis than for other Ollama tasks.

Seed — bootstrap intelligence

If you have existing items in the database but haven't configured projects yet, Seed analyses a sample of them and proposes project names, keywords, and watch topics. Review and edit the suggestions, then click Apply to save them to Settings and optionally kick off a full re-analysis.

Email ingestion (Outlook / Thunderbird)

Email is fed in from a sidecar script running on the host machine. Run the appropriate script on a schedule (e.g. Windows Task Scheduler or cron every 30 minutes):

Windows
python scripts/outlook_sidecar.py --setup once, then schedule the normal run.
Ubuntu
python scripts/thunderbird_sidecar.py via cron.

The API deduplicates by message ID so re-running is safe.

Scan progress

While a scan or re-analysis is running, the progress bar reflects how many items have been processed versus the total. The status message shows Analyzing N/Total items — ~X min remaining once at least one item has completed, giving a rolling estimate based on the last 10 items.

Re-analysis item ordering

When running ↺ Re-analyze, items are processed in priority order: user tier first (items directly addressed to you), then project, then topic, and finally general. Within each tier, newest items are processed first so your most urgent context is updated earliest.

Background batch processing

When ↺ Re-analyze is triggered and merLLM is reachable, re-analysis jobs are submitted to merLLM's batch queue in the background priority bucket — the lowest of merLLM's five strict-priority lanes. Real-time chat (chat), bounded foreground work (short), and feedback work spawned by other background jobs (feedback) all dispatch first; background only drains when every higher lane is empty. Results are applied automatically in the background — no action needed. The status message will read Re-analysis queued — N items sent to batch when this path is active.

Mobile layout

Below 916px (foldable-inner width) a Right Now card appears at the top of the main view with the top 3 attention-ranked items, overdue todo count, next situation follow-up, and a one-tap Scan button.

Below 768px the sidebar becomes an overlay drawer (tap the ☰ button to open), the items table is replaced by a single-column card layout, modals and the detail panel go full-screen, and interactive controls grow to comfortable tap-target sizes. Swipe right on a todo to mark it done; swipe left to delete it.

Below 480px the topbar wordmark subtitle is hidden. Below 430px (foldable phone in its folded outer display) the Seed and Re-analyze buttons are hidden from the topbar — run them from the unfolded view, the in-panel Right Now Scan button, or a desktop session.
